Critical / High Fixes

These three fixes address the most impactful issues. Fix #1 is the only one that could cause real financial harm.

Fix #1: Budget check bypassed when tier is pinned

File: extensions/model-router/routing.ts — decideRouting

Problem: The budget check is inside the else block of the if (pinnedTier) conditional. When a user pins "high" via /router pin high, the budget check is entirely skipped, allowing unbounded costs.

Approach: Move the budget check outside the if/else so it always applies regardless of pin state. The pinned tier should be respected, but the budget override should still be enforced as a safety cap.

// Current (broken):
if (pinnedTier) {
  tier = pinnedTier;
} else {
  // ... heuristics ...
  if (isBudgetExceeded && tier === "high") {
    tier = "medium"; // budget check only in else
  }
}

// Fixed:
if (pinnedTier) {
  tier = pinnedTier;
} else {
  // ... heuristics ...
}
// Budget check always applies:
if (isBudgetExceeded && tier === "high") {
  tier = "medium";
  phase = "implementation";
  reasoning = `Budget exceeded. Downgraded from high to medium tier. (Original: ${reasoning})`;
  isBudgetForced = true;
}

Fix #2: Remove wordCount shadowing

File: extensions/model-router/routing.ts — decideRouting

Problem: const wordCount is declared at the top of the function, then redeclared inside the if (rules && wordCount < 100) block. The inner declaration shadows the outer one and is redundant.

Approach: Remove the inner const wordCount = countWords(prompt) — the outer one is already computed and used in the condition.

Fix #3: resolveAvailableTier returns non-existent tier

File: extensions/model-router/routing.ts — resolveAvailableTier

Problem: When no tier exists in the profile, the function returns the preferred tier anyway. This causes buildRoutingDecision to throw.

Approach: Return undefined when no tier is available, and handle the undefined case in decideRouting by falling back to a default tier with a warning.

File: extensions/model-router/commands.ts — handleFix

Fix #7: handleDisable router-model guard

File: extensions/model-router/commands.ts — handleDisable

Approach: After parsing state.lastNonRouterModel, check that the provider is not "router". If it is, show a warning and suggest using /model directly.

Fix #9: Classifier prompt truncation

File: extensions/model-router/routing.ts — runClassifier

Approach: Instead of truncating the first 300 chars, keep the last 300 chars (where the actual task intent is more likely to be). Alternatively, use a smarter strategy: extract the last user message and use that as the primary context, with recent conversation as secondary context.

Test Coverage Gaps

The existing 48 tests cover config.ts and routing.ts core logic well, but miss significant areas:

  • loadRouterConfig — full config loading pipeline (global + project merge, normalization)
  • normalizeTierConfig — model aliases, fallbacks, reasoning, thinking levels
  • normalizeModelsMap — edge cases (missing model field, invalid contextWindow, etc.)
  • resolveContextWindow / resolveMaxOutputTokens — registry lookup fallback
  • hasImageAttachment — content part detection
  • getRecentConversationText — message slicing and filtering
  • countToolResults — role-based filtering
  • Extension lifecycle (index.ts) — session start, model select, turn end
  • Provider registration (provider.ts) — stream handling, fallback chains
  • Command handlers (commands.ts) — all subcommand paths
  • Status widget (ui.ts) — rendering edge cases

Recommendation: Add integration-style tests for the provider and command layers using mocked ExtensionContext and ExtensionAPI.
